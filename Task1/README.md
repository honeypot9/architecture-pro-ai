### **Задание 1. Анализ инструментов и инфраструктуры для RAG-решения**

#### **1. Анализ задачи и требований**

**Цель:** Разработать RAG-бот для эффективного поиска и предоставления релевантной информации из внутренних документов
компании.

**Ключевые аспекты:**

- **Конфиденциальность:** Значительная часть материалов (персональные данные, архитектурные решения, бизнес-процессы) не
  может быть вынесена в публичное облако.
- **Контроль доступа:** Необходимо разграничить доступ к документам в зависимости от роли сотрудника.
- **Стоимость:** Оптимизация совокупной стоимости владения с учетом как первоначальных затрат, так и
  эксплуатационных.

**Подход к обработке данных:**
Для обеспечения конфиденциальности выбираем **гибридную архитектуру**:

- **Локальный контур:** Векторные базы данных, модели эмбеддингов и метаданные развертываются на собственной
  инфраструктуре. Это гарантирует, что исходные тексты и их векторные представления не покидают компанию.
- **Облачная LLM (с проксированием):** Генерация ответов осуществляется через облачный API, но перед отправкой запросы
  проходят через прокси-слой, который:
    1. **Маскирует PII** (персонально идентифицируемую информацию) в извлекаемых фрагментах
    2. **Валидирует** запросы и ответы
    3. **Ведет учет** использования токенов

Таким образом, в облако передаются только безопасные, обезличенные контексты.

#### **2. Сравнительный анализ инструментов**

##### **2.1. Сравнение LLM-моделей**

| Критерий               | Локальные (Hugging Face)                                | Облачные (OpenAI / YandexGPT)                                    |
|------------------------|---------------------------------------------------------|------------------------------------------------------------------|
| **Качество ответов**   | Удовлетворительное, отстает в сложных reasoning-задачах | **Высокое**, стабильное качество генерации и понимания контекста |
| **Скорость работы**    | Зависит от hardware, задержки на загрузку моделей       | **Высокая**, отлаженная инфраструктура                           |
| **Стоимость**          | Высокий CAPEX, низкий OPEX                              | Прозрачный OPEX, выгодно при средних объемах                     |
| **Конфиденциальность** | **Полная**, данные не покидают периметр                 | **Условная**, требует мер по очистке данных                      |
| **Развертывание**      | Сложное, требует MLOps компетенций                      | Простое, API-интеграция                                          |

**Вывод:** Для баланса качества и затрат выбираем **облачную LLM (OpenAI GPT-4o)** с прокси-слоем для маскирования
конфиденциальных данных.

##### **2.2. Сравнение моделей эмбеддингов**

| Критерий                      | Локальные (Sentence-Transformers)                       | Облачные (OpenAI Embeddings)                  |
|-------------------------------|---------------------------------------------------------|-----------------------------------------------|
| **Скорость создания индекса** | Высокая на GPU, линейное масштабирование                | Очень высокая, но возможны лимиты по запросам |
| **Качество поиска**           | **Высокое** при выборе современной модели (BAAI/bge-m3) | **Эталонное**, оптимизировано для поиска      |
| **Стоимость**                 | Фиксированная (стоимость сервера)                       | Переменная (оплата за токены)                 |
| **Конфиденциальность**        | **Полная**                                              | **Нулевая**, все тексты уходят в облако       |

**Вывод:** Учитывая требования конфиденциальности и частоту переиндексации, выбираем **локальную модель (BAAI/bge-m3)**.

##### **2.3. Сравнение векторных БД**

| Критерий                            | ChromaDB                                                             | FAISS                                                                         |
|-------------------------------------|----------------------------------------------------------------------|-------------------------------------------------------------------------------|
| **Скорость поиска и индексации**    | Достаточная для средних нагрузок                                     | **Высокая**, индустриальный стандарт для ANN                                  |
| **Сложность внедрения и поддержки** | **Низкая**, простое развертывание, встроенное управление метаданными | **Высокая**, требует ручной настройки индексов и управления метаданными       |
| **Удобство в работе**               | **Высокое**, интуитивный API, встроенные функции                     | Низкое, необходимо самостоятельно реализовывать персистентность               |
| **Стоимость владения**              | Низкая стоимость внедрения                                           | Низкая стоимость инфраструктуры, но **высокая стоимость инженерного времени** |

**Вывод:** Для продакшн-решения выбираем **FAISS** как наиболее производительное и гибкое решение, несмотря на сложность
первоначальной настройки.

#### **3. Рекомендуемая конфигурация сервера**

Для выбранной гибридной схемы оптимальна конфигурация:

- **CPU:** 16+ vCPU (Intel Xeon / AMD EPYC)
- **RAM:** 64+ ГБ
- **GPU:** 1 x NVIDIA RTX 4000 (16+ ГБ VRAM) — для ускорения создания эмбеддингов
- **SSD:** 1+ ТБ NVMe

Данная конфигурация обеспечивает высокую скорость индексации и работы с векторной БД, при этом не требуется мощных GPU
для инференса.

### **Финальное решение**

- **LLM:** YandexGPT
- **Модель эмбеддингов:** BAAI/bge-m3 (локально)
- **Векторная БД:** FAISS

**Обоснование:**

1. **Безопасность:** Конфиденциальные данные остаются в локальном контуре. В облако передаются только очищенные от PII
   фрагменты.
2. **Экономическая эффективность:** Избегаем повторяющихся затрат на облачные эмбеддинги. Платим только за генерацию
   ответов.
3. **Качество:** Сочетание современной модели эмбеддингов (bge-m3) и мощной LLM (YandexGPT) обеспечивает точный поиск и
   качественные ответы.
4. **Производительность:** FAISS обеспечивает высокую скорость поиска даже на больших объемах данных.

Данный подход обеспечивает сбалансированное решение, отвечающее требованиям безопасности, качества и стоимости.
